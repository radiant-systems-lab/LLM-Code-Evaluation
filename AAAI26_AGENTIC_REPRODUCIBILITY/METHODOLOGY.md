# Agentic AI Code Reproducibility: Research Methodology

## 1. Research Problem

### 1.1 The Reproducibility Crisis in AI-Generated Code

Modern Large Language Models (LLMs) with agentic capabilities—those equipped with command-line interface (CLI) tools and the ability to execute multi-step workflows—can generate complete software projects including source code, dependency specifications, and documentation. These "agentic AI" systems (such as Claude Code, Gemini Code Assist, and GitHub Copilot with Codex) promise to accelerate software development by automating routine coding tasks.

However, a critical question remains unanswered: **Are the dependency specifications generated by agentic AI systems accurate and complete enough to ensure reproducibility?**

Reproducibility is a cornerstone of software engineering. A project is reproducible if:
1. Its dependency specifications are complete (all required dependencies are listed)
2. Its dependency specifications are precise (no unnecessary dependencies are listed)
3. The code can execute successfully in a clean environment using only the specified dependencies

### 1.2 The Core Problem

When agentic AI systems generate code, they typically produce:
- **Source code** (e.g., Python scripts)
- **Dependency specifications** (e.g., `requirements.txt` files)
- **Documentation** (e.g., README files with usage instructions)

The fundamental problem we investigate is: **Do LLMs accurately specify all dependencies needed for their generated code to run, or do they suffer from "dependency amnesia"—failing to list critical packages while sometimes including unnecessary ones?**

This has serious implications:
- **For developers**: Incomplete dependencies lead to frustrating "works on my machine" scenarios
- **For research**: Non-reproducible code undermines scientific validity
- **For production**: Incomplete dependency specifications create security vulnerabilities and deployment failures

### 1.3 Research Questions

This study addresses three primary research questions:

**RQ1: Completeness** - Do agentic AI systems specify all dependencies required for their generated code to execute?

**RQ2: Precision** - Do agentic AI systems avoid specifying unnecessary dependencies?

**RQ3: Iterative Fixing** - When dependencies are incomplete, can they be identified and fixed through iterative debugging, and what types of errors emerge (missing dependencies vs. code bugs)?

## 2. Research Approach

### 2.1 Comparative Analysis Framework

We employ a **comparative case study** methodology analyzing three leading agentic AI systems:

1. **Claude Code** (Anthropic) - CLI tool with function-calling capabilities
2. **Gemini Code Assist** (Google) - CLI tool with agentic features
3. **GitHub Copilot with Codex** (OpenAI) - CLI interface to GPT-4 with code generation

### 2.2 Prompt-Based Generation

For each LLM, we use **40 standardized prompts** designed to generate realistic software projects across various domains:
- Web scraping
- Data analysis and visualization
- Machine learning applications
- API development
- Text processing
- Image processing
- System utilities

Each prompt explicitly requests:
```
IMPORTANT: Provide EVERYTHING needed for reproduction:
- Complete working [Language] code
- Complete requirements.txt with ALL dependencies and versions
- Brief usage instructions

Make it 100% reproducible in a clean environment.
```

This ensures all three LLMs receive identical instructions, enabling fair comparison.
### 2.3 Three-Tier Dependency Analysis

We analyze dependencies at three distinct levels:

**Tier 1: Claimed Dependencies**
- What the LLM explicitly specifies in `requirements.txt`
- Represents the LLM's understanding of required packages

**Tier 2: Working Dependencies**
- What's actually required after manual iterative fixing
- Includes both claimed dependencies + any missing packages discovered through debugging
- Represents ground truth for execution

**Tier 3: Runtime Dependencies**
- What SciUnit captures during actual execution
- Includes all transitively imported packages
- Represents comprehensive dependency footprint

By comparing these three tiers, we identify:
- **Completeness gaps**: Working deps - Claimed deps (what the LLM forgot)
- **Precision gaps**: Claimed deps - Runtime deps (what the LLM unnecessarily included)
- **Runtime gaps**: Runtime deps - Claimed deps (transitive dependencies)

#### 3.1.3 SciUnit Installation

SciUnit is a reproducibility capture tool developed by the Radiant Systems Lab that instruments Python execution to capture runtime dependencies.

```bash
cd /home/ubuntu/agentic_reproducibility
git clone https://github.com/radiant-systems-lab/sciunit.git
cd sciunit
pip3 install --user -e .
```

SciUnit provides two key capabilities:
1. **Workspace creation**: Isolated environments for each execution
2. **Dependency capture**: Records all packages imported during runtime

### 3.2 Manual Iterative Processing Methodology

**Core Principle**: This research employs **manual, iterative processing** rather than batch automation. This is critical because:

1. **Errors vary unpredictably** - Some projects have missing dependencies, others have code bugs, some need system packages
2. **Human judgment required** - Distinguishing between dependency issues vs. code bugs requires contextual understanding
3. **Research validity** - Manual processing ensures we understand each failure mode rather than masking issues with automated fixes
4. **Realistic simulation** - This mirrors how developers actually debug LLM-generated code


#### 3.2.1 The Seven-Step Processing Workflow

For each of the 40 projects per LLM (120 total), we follow this workflow:

---

**STEP 1: Read and Understand**

```bash
# Navigate to project directory
cd /home/ubuntu/agentic_reproducibility/{llm}_generated/python/p_{X}

# Read the prompt that generated this code
cat README.md | head -20  # Prompts usually documented in README

# Examine the generated code
cat {script_name}.py

# Read the claimed dependencies
cat requirements.txt
```

**Objective**: Understand what the project is supposed to do and what dependencies the LLM claims are needed.

---

**STEP 2: Install Claimed Dependencies**

```bash
pip3 install -r requirements.txt --quiet
```

**Objective**: Install exactly what the LLM specified—no more, no less. This is the LLM's "claim" about what's needed.

**Handling Special Cases**:
- If `requirements.txt` is empty → LLM claims code uses only standard library
- If versions are pinned (e.g., `pandas==2.1.3`) → Install exact versions
- If no versions specified → Allow pip to resolve latest compatible versions

---

**STEP 3: Attempt Execution**

```bash
python3 {script_name}.py
```

**Objective**: Try to run the script with only the claimed dependencies installed.

**Possible Outcomes**:
- ✅ **Success** → Proceed to STEP 4
- ❌ **Import Error** → Missing dependency (proceed to STEP 3a)
- ❌ **Code Error** → Bug in generated code (proceed to STEP 3b)
- ❌ **System Error** → Missing system library or version conflict (proceed to STEP 3c)

---

**STEP 3a: Fix Missing Dependencies (Completeness Gap)**

If an import error occurs:

```python
ImportError: No module named 'pandas'
```

**Action**:
1. Identify the missing package
2. Install it: `pip3 install pandas`
3. Document as **completeness gap** (LLM forgot to specify this)
4. Return to STEP 3 and re-attempt execution
5. Repeat iteratively until all import errors are resolved

**Example**:
```
Claimed: opencv-python-headless==4.9.0.80
Error: numpy.core.multiarray failed to import (numpy 2.2.6 incompatible)
Fix: pip3 install 'numpy<2'
Gap: LLM specified opencv but forgot the numpy<2 constraint
```

---

**STEP 3b: Fix Code Bugs**

If a code error occurs:

```python
SyntaxError: invalid syntax at line 38
```

**Action**:
1. Analyze the error
2. Identify the bug in LLM-generated code
3. Create a fixed version in `/tmp/fix_pX.py`
4. Test the fix: `python3 /tmp/fix_pX.py`
5. Document as **code bug** (not a dependency issue)
6. Use the fixed version for SciUnit capture

**Important Distinction**:
- **Completeness gap**: LLM forgot to list a dependency
- **Code bug**: LLM wrote syntactically/logically incorrect code
- These are tracked separately in results

---

**STEP 3c: Handle System Dependencies**

Some Python packages require system libraries or specific constraints:

```bash
# Example: opencv requires numpy<2 for compatibility
pip3 uninstall -y numpy
pip3 install 'numpy<2'

# Example: Some packages need system libraries
sudo apt-get install -y libsm6 libxext6
```

**Objective**: Resolve system-level requirements while documenting what was needed.

---

**STEP 4: Capture Runtime Dependencies with SciUnit**

Once the script runs successfully, capture its actual dependency footprint:

```bash
# Create isolated SciUnit workspace
cd /home/ubuntu/sciunit
/home/ubuntu/.local/bin/sciunit create {llm}_p{X}_workspace

# Execute script under SciUnit instrumentation
cd /home/ubuntu/sciunit/{llm}_p{X}_workspace
/home/ubuntu/.local/bin/sciunit exec {script_name} -- \
  python3 /home/ubuntu/agentic_reproducibility/{llm}_generated/python/p_{X}/{script_name}.py

# Mark execution as complete
/home/ubuntu/.local/bin/sciunit repeat last mark-complete
```

**What SciUnit Captures**:
- All imported packages (including transitive imports)
- File system reads/writes
- Network activity
- System calls

**Objective**: Understand the **complete** dependency footprint during actual execution, not just what pip installed.

---

**STEP 5: Export and Analyze Runtime Dependencies**

```bash
# Export captured dependencies
cd /home/ubuntu/sciunit/{llm}_p{X}_workspace
/home/ubuntu/.local/bin/sciunit export-requirements

# Examine runtime dependencies
cat e1-requirements.txt
```

**Analysis**:
- Compare runtime deps vs. claimed deps
- Identify transitive dependencies (imported but not explicitly installed)
- Calculate precision gap (unnecessary claimed deps)

---

**STEP 6: Document Results in CSV**

Append comprehensive analysis to the results CSV with 20 columns documenting all aspects of the project including prompt, dependencies (claimed/working/runtime), execution success, error types, gaps, and detailed notes.

---

**STEP 7: Clean Environment and Verify Baseline**

**Critical Step**: Restore environment to exactly 91 packages:

```bash
# Uninstall all project-specific packages
pip3 uninstall -y package1 package2 package3 ...

# Verify baseline restoration
pip3 list | wc -l
# MUST return 91
```

**Objective**: Ensure no cross-contamination between projects. Each project must start with a pristine environment.

#### 3.2.2 Why Manual Processing is Essential

**Batch Automation Fails Because**:

1. **Diverse error types require different fixes**:
   - Missing dependency → `pip3 install package`
   - Code syntax error → Edit source code
   - Version conflict → `pip3 install 'package<version'`
   - System library → `apt-get install library`

2. **Context matters for classification**:
   - Is this a dependency gap or code bug?
   - Is this package necessary or transitive?
   - Should we fix or document as LLM failure?

3. **Research integrity**:
   - Manual processing ensures we understand failure modes
   - Automated fixes might mask systematic LLM issues
   - Human judgment required for ambiguous cases

4. **Realistic developer experience**:
   - This mirrors how developers actually debug LLM code
   - Shows the true "hidden cost" of incomplete dependencies

### 3.3 Data Collection and Metrics

#### 3.3.1 Primary Metrics

For each project, we collect:

**Dependency Counts**:
- `claimed_count`: Number of dependencies in `requirements.txt`
- `working_count`: Number after iterative fixing
- `runtime_count`: Number captured by SciUnit

**Gap Metrics**:
- `completeness_gap = working_count - claimed_count`
  - How many dependencies the LLM forgot
  - 0 = perfect completeness
  - >0 = LLM underspecified

- `precision_gap = claimed_count - runtime_count`
  - How many unnecessary dependencies the LLM included
  - 0 = perfect precision
  - >0 = LLM overspecified

- `runtime_gap = runtime_count - claimed_count`
  - Transitive dependencies not explicitly listed
  - Shows dependency closure complexity

**Error Classification**:
- `None`: No errors, code ran successfully with claimed deps
- `ImportError`: Missing dependency (completeness gap)
- `CodeBug-Fixed`: Syntax or logic error in generated code
- `SystemError`: Version conflict or system library needed

#### 3.3.2 Qualitative Data

**Detailed Notes** (CSV column 20):
- Description of all iterative fixes applied
- Rationale for classifying errors
- Observations about LLM behavior
- Special cases (empty requirements.txt for stdlib-only code)

#### 3.3.3 Special Cases

**Empty requirements.txt**:
- Some projects correctly have empty `requirements.txt`
- These use only Python standard library
- Document as "Correct empty requirements.txt (stdlib only)"

**Large Installations**:
- Some projects require heavy packages (e.g., TensorFlow 475MB)
- Document installation size and time
- Verify complete cleanup afterward

**Network-Dependent Code**:
- Web scrapers may fail due to URL availability
- Document as "URLs blocked but imports work"
- Focus on dependency correctness, not runtime output

### 3.6 Limitations and Threats to Validity

#### 3.6.1 Internal Validity

**Researcher Bias**:
- Manual processing involves human judgment in fixing errors
- Mitigation: Document all decisions in CSV notes
- Different researchers might fix code differently

**Environment Variability**:
- Network conditions affect package downloads
- Ubuntu version affects system packages
- Mitigation: Use consistent EC2 instance throughout

#### 3.6.2 External Validity

**Prompt Selection**:
- 40 prompts may not represent all code generation scenarios
- Focused on common software engineering tasks
- Generalization limited to similar project types

**Language Scope**:
- This methodology focuses on Python
- Results may not generalize to other languages (JavaScript, Java, etc.)

**LLM Version**:
- LLMs update frequently
- Results valid for specific LLM versions tested
- Future versions may improve dependency specification

#### 3.6.3 Construct Validity

**What is "Correct"?**:
- Working dependencies represent "ground truth"
- But developers might have fixed differently
- SciUnit captures runtime deps, but some packages imported conditionally

**Reproducibility Definition**:
- We define reproducibility as "executes without errors"
- Doesn't guarantee functional correctness
- Code might run but produce wrong results

- Complete methodology documented

## 5. Summary of Methodology

This research employs a **rigorous, manual, iterative methodology** to evaluate dependency specification accuracy in agentic AI-generated code:

1. **40 standardized prompts** × **3 LLMs** = **120 projects analyzed**

2. **Seven-step processing workflow** for each project:
   - Read & understand
   - Install claimed dependencies
   - Execute & debug iteratively
   - Capture runtime dependencies with SciUnit
   - Export & analyze
   - Document in CSV
   - Clean environment & verify baseline

3. **Three-tier dependency analysis**:
   - Claimed (LLM specification)
   - Working (after manual fixing)
   - Runtime (SciUnit capture)

4. **Comprehensive metrics**: Completeness gaps, precision gaps, error types, qualitative notes

5. **Strict quality assurance**: Environment hygiene, data integrity validation, reproducibility verification

This methodology provides a **realistic, human-in-the-loop evaluation** of how agentic AI code performs when developers actually try to use it—exposing the true cost of incomplete dependency specifications.
